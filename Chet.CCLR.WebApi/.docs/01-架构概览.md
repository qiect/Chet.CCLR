# 架构概览

## 1. 文档概述

本文档详细描述了Chet.CCLR.WebApi项目的架构设计，包括分层架构、各层职责、组件关系以及依赖引用规则。本架构采用Clean Architecture和Domain-Driven Design (DDD)模式，旨在实现高内聚、低耦合，便于维护和扩展。

## 2. 架构目标

- **解耦**: 各层之间通过依赖注入实现松耦合
- **可测试性**: 便于编写单元测试和集成测试
- **可扩展性**: 支持单体服务和微服务架构
- **可维护性**: 清晰的分层结构便于理解和维护
- **可重用性**: 核心业务逻辑可在不同部署环境中重用
- **安全性**: 内建安全机制，符合安全最佳实践

### 3.2 核心层 (Core Layer)

**职责**: 包含领域模型、接口定义和共享类，是整个应用的核心，不依赖任何其他层。

#### 3.2.1 Domain子层

**职责**: 定义领域模型和业务实体，体现业务核心概念。

**组件**: 
- `BaseEntity.cs`: 所有实体的基类，包含通用字段如Id、CreatedAt、UpdatedAt
- `User.cs`: 用户实体，包含用户的基本信息

**特点**: 
- 只包含实体定义，不包含业务逻辑
- 实体之间的关系通过导航属性定义
- 遵循领域驱动设计原则

#### 3.2.2 Contracts子层

**职责**: 定义所有业务接口，包括仓储接口、服务接口等。

**组件**: 
- `IRepository<T>`: 通用仓储接口，定义CRUD操作
- `IUserRepository`: 用户仓储接口，扩展通用仓储接口
- `IUserService`: 用户服务接口，定义用户相关业务逻辑
- `IAuthService`: 认证服务接口，定义认证相关业务逻辑
- `IJwtService`: JWT服务接口，定义JWT令牌相关操作
- `ICacheService`: 缓存服务接口，定义缓存操作

**特点**: 
- 只定义接口，不包含实现
- 接口设计遵循单一职责原则
- 便于测试和替换实现

#### 3.2.3 Shared子层

**职责**: 包含各层共享的类和异常定义。

**组件**:
- `ApiResponse.cs`: 通用API响应包装器
- `ErrorResponse.cs`: 错误响应模型
- `PaginatedResponse.cs`: 分页响应模型
- `NotFoundException.cs`: 未找到异常
- `BadRequestException.cs`: 错误请求异常

**特点**:
- 不依赖任何其他层
- 提供跨层共享的数据结构和异常类型

### 3.3 应用层 (Application Layer)

**职责**: 包含业务逻辑、数据传输对象(DTO)和对象映射配置。

#### 3.3.1 Services子层

**职责**: 实现业务逻辑，协调不同组件完成业务功能。

**组件**:
- `UserService.cs`: 用户业务逻辑实现
- `AuthService.cs`: 认证业务逻辑实现
- `JwtService.cs`: JWT令牌业务逻辑实现

**特点**:
- 依赖于Core层的接口定义
- 不直接依赖Infrastructure层的具体实现
- 通过依赖注入获取所需的接口实现

#### 3.3.2 DTOs子层

**职责**: 定义数据传输对象，用于API层与应用层之间的数据传递。

**组件**:
- `UserDto.cs`: 用户数据传输对象
- `UserCreateDto.cs`: 用户创建数据传输对象
- `UserUpdateDto.cs`: 用户更新数据传输对象
- `LoginDto.cs`: 登录数据传输对象
- `RegisterDto.cs`: 注册数据传输对象
- `JwtTokenDto.cs`: JWT令牌数据传输对象

**特点**:
- 不包含业务逻辑
- 专用于数据传输
- 有助于API版本管理和数据验证

#### 3.3.3 Mapping子层

**职责**: 配置对象映射规则，实现DTO与实体之间的转换。

**组件**:
- `MappingProfile.cs`: AutoMapper配置文件

**特点**:
- 集中管理对象映射配置
- 减少手动映射代码
- 提高开发效率

### 3.4 基础设施层 (Infrastructure Layer)

**职责**: 提供具体的技术实现，包括数据访问、缓存、日志等功能。

#### 3.4.1 Data子层

**职责**: 实现数据访问逻辑，包括数据库上下文和仓储模式实现。

**组件**:
- `AppDbContext.cs`: EF Core数据库上下文
- `EfCoreRepository.cs`: 通用仓储实现
- `UserRepository.cs`: 用户仓储实现

**特点**:
- 依赖EF Core ORM
- 实现Core层定义的仓储接口
- 处理具体的数据库操作

#### 3.4.2 Caching子层

**职责**: 提供缓存实现，支持分布式缓存和内存缓存。

**组件**:
- `RedisCacheService.cs`: Redis缓存服务实现
- `NoOpCacheService.cs`: 空操作缓存服务（用于测试）

**特点**:
- 实现Core层定义的缓存接口
- 支持多种缓存后端
- 提供缓存策略配置

#### 3.4.3 Logging子层

**职责**: 配置和实现日志记录功能。

**组件**:
- `SerilogExtensions.cs`: Serilog配置扩展

**特点**:
- 集成Serilog日志框架
- 支持多种输出目标
- 提供结构化日志记录

#### 3.4.4 Configuration子层

**职责**: 管理应用配置和设置。

**组件**:
- `AppSettings.cs`: 应用配置模型

**特点**:
- 强类型的配置模型
- 支持多环境配置
- 配置验证和默认值

### 3.5 API层 (表示层)

**职责**: 处理HTTP请求和响应，提供RESTful API端点。

#### 3.5.1 Controllers子层

**职责**: 定义API端点，处理HTTP请求。

**组件**:
- `UsersController.cs`: 用户相关API端点
- `AuthController.cs`: 认证相关API端点

**特点**:
- 通过依赖注入获取服务
- 负责HTTP请求/响应处理
- 不包含业务逻辑

#### 3.5.2 Middleware子层

**职责**: 实现中间件功能，如异常处理、认证等。

**组件**:
- `ErrorHandlingMiddleware.cs`: 全局异常处理中间件

**特点**:
- 横切关注点处理
- 统一错误响应格式
- 与业务逻辑解耦

## 4. 依赖关系

### 4.1 依赖方向
- API层 -> Application层
- Application层 -> Core层
- Infrastructure层 -> Core层
- Tests层 -> 所有其他层

### 4.2 依赖注入配置
依赖注入配置在`Program.cs`中进行，通过扩展方法组织不同功能模块的配置。

## 5. 部署架构

### 5.1 单体架构
所有功能模块部署在单一应用中，适合中小规模应用。

### 5.2 微服务架构
功能模块独立部署，通过API网关通信，适合大规模分布式应用。

### 5.3 容器化部署
使用Docker容器化部署，支持Kubernetes编排。

## 6. 安全考虑

### 6.1 认证与授权
- JWT令牌认证
- 基于角色的访问控制
- 密码安全存储（BCrypt）

### 6.2 数据保护
- 输入验证
- SQL注入防护
- XSS防护

## 7. 性能优化

### 7.1 缓存策略
- Redis分布式缓存
- HTTP响应缓存
- 内存缓存

### 7.2 数据库优化
- 查询优化
- 索引策略
- 连接池配置

### 7.3 异步处理
- 异步API端点
- 非阻塞I/O操作
